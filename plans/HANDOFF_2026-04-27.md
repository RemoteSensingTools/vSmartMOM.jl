Variables: `wavenumbers(2003)` ν ∈ [-20, 20000] cm⁻¹ uniform 10 cm⁻¹; `self_absco_ref`, `for_absco_ref`, `self_texp`; scalars `ref_press = 1013` mbar, `ref_temp = 296` K.
- **New file**: [src/Absorption/mtckd.jl](file:///home/sanghavi/code/github/uni_vSmartMOM/src/Absorption/mtckd.jl). Exports `MTCKDTable`, `MTCKDBand`, `load_mtckd`, `build_mtckd_band`, `compute_τ_h2o_continuum!`. Implementation follows standard LBLRTM convention:
  - `σ_self(ν,T,p_h2o) = C_self_ref(ν) · radterm(ν,T) · (p_h2o/p_ref) · (T_ref/T)^texp(ν)`
  - `σ_for(ν,T,p_dry)  = C_for_ref(ν)  · radterm(ν,T) · (p_dry/p_ref)`
  - `radterm(ν,T) = ν · tanh(c₂ ν / 2T)`, `c₂ = 1.4388 cm·K`
  - τ_layer = (σ_self + σ_for) · n_h2o · Δz_cm. Float64 internally; outside the file's [-20, 20000] cm⁻¹ range → 0.
- `mtckd_file::String` added to `AbsorptionParameters`. Parser added. Hooked into both forward variants and the linearized variant of `model_from_parameters`. `mtckd_file:` path added to both EMIT YAMLs.

## ⚠️ Known bug — fix this first tomorrow

**Symptom**: MT_CKD smoke test errored with
```
BoundsError: attempt to access 54-element Vector{Float32} at index [55]
in compute_atmos_profile_fields at src/CoreRT/tools/atmo_prof.jl:60
```

**Cause** (best guess): the YAML's `T` and `p` arrays are both length 54, but `compute_atmos_profile_fields` treats `T` as **per-layer** (`n_layers = length(T)`) and `p` as **interfaces** (expects `n_layers + 1`). H₂O `vmr` array is correctly length 53 (per-layer). With T=54 the loop tries `p_half[55]` and crashes.

**Mystery**: the **CIA smoke test earlier in the session ran successfully** with the same setup (printed `nlayers=53`). Either (a) the precompile picked up an old definition between the two tests, or (b) something I changed in `types.jl` between the two tests altered the function's behavior. **Investigate before assuming a fix is required.**

**Likely fix**: trim the YAML `T` and `q` arrays to 53 entries (per-layer values, e.g. evaluated at layer mid-pressures `(p[i]+p[i+1])/2`). The new T list I generated for Phase D has 54 values; needs to be regenerated as 53.

The full forward-RT pipeline in `emit_modtran_noRS_scenarios.jl` calls `model_from_parameters` (which calls `compute_atmos_profile_fields` internally) and was working **before** Phase D. So either the Phase D edit broke it, or the bug is dormant in some code path that the regression script avoids. **First step tomorrow: re-run the CIA smoke test to confirm it still works post-Phase-D, then bisect.**
