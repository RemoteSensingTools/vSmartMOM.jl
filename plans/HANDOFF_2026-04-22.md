## Root Cause Found (Final Session)

**ALL previous BoundsErrors were caused by a Float32/Float64 precision issue in the wavenumber axis, NOT by the atmospheric pressure profile.**

### The Bug

```
spec_bands: (1e7/2510.0):0.1:(1e7/380.0)
```

`1e7/2510.0` in Float64 = **3984.063745019920** (the LUT lower bound)  
`Float32(3984.063745...)` = **3984.063720703125** (rounds DOWN by 2.4e-5)

The Interpolations.jl `BSpline(Cubic(Line(OnGrid())))` bounds check:
```
lbound = first(ν_grid) = 3984.063745019920  (Float64)
query  = 3984.0637f0   = 3984.063720703125  (Float32 → Float64)
3984.063745 <= 3984.063720  →  FALSE  →  BoundsError
```

Every single run failed at the very first wavenumber in the grid.

### The Fix (Already Applied)

Both YAML files now have:
```yaml
spec_bands:
  - 3984.2:0.1:26315.7
```

This starts 0.14 cm⁻¹ above the LUT lower bound (2509.9 nm instead of 2510.0 nm — negligible physical difference).

## Archaeology: Why Previous Attempts Failed

The atmospheric profile was repeatedly modified (38 layers → 24 layers → 53 layers) 
trying to fix the pressure bounds issue. This was a **red herring** — the pressure 
values like `30.5f0` and `30.01f0` were VALID for the LUT pressure domain [0.01, 1080.01].
The actual failing check was always the wavenumber axis: `3984.0637f0 < 3984.06374501992`.
The error message just happened to print all three indices together, obscuring the real culprit.

The LUT's atmospheric profile alignment work is harmless (the 53-layer profile is fine
and well-aligned), but it was not the blocking issue.
